# Manual QA / UX-аудит: оба flow Briefing Studio

Ручной сценарий приёмки для wizard-flow и brand-aware freeform-flow. Рассчитан на
проверку **без реального LLM** — заполненные и pre-generated состояния создаёт seed-скрипт.
Где AI-операции требуют ключа (analyze/generate/summarize/structure/…), это отмечено как
«нужен `LLM_API_KEY`» — такие шаги опциональны.

## 0. Предусловия

- Backend и frontend запущены (Docker `docker compose up --build`, либо локально:
  `uvicorn app.main:app --reload` + `npm run dev`). См. README.
- Миграции применены (`alembic upgrade head`, head = `0006`).
- Frontend: http://localhost:5173 · Backend: http://localhost:8000 (Swagger `/docs`).

## 2. Проверка wizard-flow

1. Открыть `/briefs` → найти `[DEMO] Wizard — промо-ролик` (без бейджа Freeform) → открыть.
2. Должен открыться wizard-редактор `/brief/:id` (НЕ `/review`).
3. Пройти по шагам: поля предзаполнены (тип задачи, цель, сообщения, тон, must-have и т.д.).
4. Справа **Live brief**: секции в состоянии filled/filling, прогресс и confidence считаются
   на фронте из `context_json`.
5. На шаге предпросмотра — документ из 17 разделов (не JSON).
6. (нужен `LLM_API_KEY`) «Проанализировать бриф» и «Сгенерировать бриф».
7. Изменить любое поле → проверить, что автосейв срабатывает (индикатор «Сохранено»).

**Успех:** wizard открывается в редакторе, поля заполнены, Live brief отражает контекст,
предпросмотр рендерится как документ, автосейв работает.

## 3. Проверка brand-aware freeform-flow

1. `/brands` → есть `[DEMO] Север` → открыть `/brands/:id`, виден `brand_context_json`.
2. Создать новый freeform-бриф: с `/brands/:id` кнопка → `/brief/new/freeform?brand=:id`
   (либо `/brief/new/freeform` и выбрать бренд).
3. **Выбор структуры итогового брифа** (новый блок до свободного ввода):
   - **Стандартная структура** (по умолчанию): загружается `GET .../template/default` — дерево
     из ~12 разделов с полями и бейджами «обязательное». Проверить, что дерево отрисовано сразу.
   - **Из референса**: переключить режим → вставить текст-референс структуры → «Декомпозировать
     структуру» (`POST .../template/decompose`; нужен `LLM_API_KEY`). Дерево заменяется на
     разобранную структуру. Без ключа — остаётся стандартная (ошибка показывается, не блокирует).
   - Снять/поставить чекбоксы разделов/полей; при снятом разделе его поля становятся неактивны.
4. Вставить клиентский текст → «Создать и сделать summary» → редирект на `/brief/:id/review`.
   Выбранная структура сохраняется в брифе (`selected_template_json`).
   - (нужен `LLM_API_KEY` для реального summary; без ключа используйте готовый
     `[DEMO] Freeform — на ревью` из seed для остального осмотра.)
5. На review-экране для `[DEMO] Freeform — на ревью` экран **пошаговый** (один активный шаг в
   StepPanel, навигация через ReviewStepper — см. блок «Review-stepper, сводка и loaders» ниже).
   5 шагов:
   1. **Структура документа** — дерево разделов/полей с чекбоксами; «Сохранить структуру»
      (`POST .../select-template`). У брифа без шаблона — CTA «Использовать стандартную структуру».
   2. **Исходный бриф + Summary** — исходный текст (collapsible) + `input_summary_json` +
      «Подтвердить summary».
   3. **Структурирование** — поля с бейджами **source_type / status / confidence**.
   4. **Уточнения** — вопросы `clarifications` (importance critical/recommended/optional) + ответы.
   5. **Финальный бриф / Export** — генерация + markdown + Copy / Download MD / Download JSON /
      Download DOCX.
6. (нужен `LLM_API_KEY`) Ответить на уточнения → «Применить» → «Сгенерировать финальный бриф».
7. 409-гейт: если в структуре есть `critical_missing`/`conflict` (в т.ч. `required`-поле шаблона
   без данных), generate-final возвращает понятную ошибку «Не подтверждены критичные поля: …».

**Успех:** структура выбирается до ввода (default/референс), сохраняется в бриф; freeform-бриф
ведёт на `/review`; пошаговый flow проходит структура → summary → поля → уточнения → финал;
бейджи читаемы; generate-final гейтится при критичных полях.

### Review-stepper, сводка и loaders

**ReviewStepper** (сверху):
- активный шаг подсвечен; клик по шагу переключает активный шаг;
- кнопки «← Назад / Далее →» в подвале панели двигают по шагам;
- статусы шагов: `done` (✓), `ready` (◆), `needs_action` (!), `blocked` (🔒), `processing` (⟳);
  напр. у `[DEMO] Freeform — на ревью` шаги 1–4 `done`, «Финальный бриф» `ready` и активен.

**ReviewStateSummary** (между шапкой и степпером):
- прогресс «N/5 шагов готово»;
- чипы: «Структура выбрана / Стандартная структура», «Summary подтверждён / не подтверждён»,
  «Блокеров: N» (если есть), «Документ актуален / устарел / Не сгенерирован», ссылка «Бренд».

**Loaders** (нужен `LLM_API_KEY`, чтобы увидеть в действии):
- внутри активного шага показывается `ProcessingState` (спиннер + текст): summarize →
  «Готовим summary…», structure → «Структурируем бриф…», clarify → «Формируем уточняющие
  вопросы…», apply → «Применяем ответы…», generate final → «Генерируем итоговый документ…»;
  на это время кнопки шага disabled, шаг в степпере — `processing`;
- export markdown/json/**docx** — лёгкий busy на самой кнопке («Скачиваем…»), без большой панели.

**Export DOCX** (шаг 5 / wizard-зона):
- кнопка **Download DOCX** есть и в review (шаг «Финальный бриф / Export»), и у wizard
  generated-брифа (блок AiActions);
- клик → скачивается `brief-{id}.docx`; на время — busy «Скачиваем…» (review);
- файл открывается в Word / LibreOffice / Pages (или хотя бы распаковывается как zip:
  внутри `word/document.xml`, `[Content_Types].xml`);
- если `generated_markdown` отсутствует — `GET /export/docx` → **409** (кнопка показывается
  только для сгенерированного брифа).

## 4. Pre-generated документы, версии, export

Открыть `[DEMO] Wizard — готовый бриф` и `[DEMO] Freeform — готовый бриф`:

1. Виден готовый markdown-документ с бейджем «AI-generated».
2. **Versions**: раскрывается история (минимум v1) в документном стиле.
3. **Export**: «Download Markdown» (`/export/markdown`), «Download JSON» (`/export/json`) и
   «Download DOCX» (`/export/docx`) скачивают файлы; до генерации md/docx-export дают 409
   (на готовом — 200). DOCX открывается в Word/LibreOffice/Pages (или как zip).
   Во время скачивания кнопка показывает «Скачиваем…», остальные действия disabled (freeform на
   шаге «Финальный бриф / Export»).
4. **Copy** копирует markdown в буфер.
5. `is_generated_outdated`:
   - У готовых брифов сразу — НЕ устарел (hash совпадает с источником `structured + template`).
   - Изменить источник → документ помечается «Требует перегенерации», в списке `/briefs` бейдж
     «Требует перегенерации». Способы:
     - freeform: изменить и **сохранить структуру** (секция «Структура итогового брифа» на
       `/review`) — для template-aware брифа смена шаблона делает документ устаревшим;
     - freeform: изменить structured-поля; wizard: любое поле контекста.
6. У `[DEMO] Freeform — готовый бриф` версия-snapshot содержит `{structured, template}`, а
   `generation_meta_json.template_source = "reference"` (видно в `/export/json` или Swagger).

**Успех:** документ, версии и export работают офлайн; устаревание загорается после правки
структуры или шаблона.

## 5. Что считать успешным прохождением

- Оба flow доходят до конца без 5xx и без падений UI.
- Wizard-брифы открываются в редакторе, freeform-брифы — на `/review`.
- В списке `/briefs` freeform помечены бейджем и ведут на `/review`; wizard — на `/brief/:id`.
- Pre-generated брифы показывают markdown, версии и экспортируются (md/json/docx);
  `is_generated_outdated`
  корректно реагирует на изменение источника.
- Удаление бренда не ломает связанные брифы (их `brand_id` обнуляется).
- Seed идемпотентен: повторный запуск не плодит дубли.

## 6. UX-долги / что проверить глазами

Открытые вопросы для визуального аудита (фиксировать наблюдения, не блокеры приёмки):

- [ ] **Два режима**: понятно ли с главной `/`, что есть два способа создать бриф?
- [ ] **Отличие brand-aware от wizard**: ясно ли пользователю, чем freeform отличается от
      пошагового wizard, и когда что выбирать?
- [ ] **Создание бренда**: легко ли найти путь «создать бренд» до создания freeform-брифа?
- [ ] **Подтверждение summary**: очевидно ли, что summary нужно подтвердить перед структурой?
- [ ] **Бейджи source/status/confidence**: читаются ли без легенды? Понятна ли разница
      `confirmed` / `confirmed_by_brand` / `needs_confirmation` / `*_missing` / `conflict`?
- [ ] **409 на generate-final**: понятно ли из текста ошибки, ПОЧЕМУ нельзя сгенерировать и
      что именно подтвердить?
- [ ] **Clarifications**: удобно ли отвечать на вопросы (особенно много вопросов)? Видна ли
      важность (critical/recommended/optional)?
- [ ] **Редактирование structured fields**: хватает ли того, что поля нельзя править инлайн
      (правка только через clarifications)? Это осознанный долг MVP — оценить остроту.
- [ ] **Template editor — чекбоксы**: понятно ли, что снятие галочки с раздела отключает его
      поля? Очевидна ли иерархия раздел → поля?
- [ ] **Template editor — required**: заметны ли бейджи «обязательное»? Понятно ли, что
      обязательное поле без данных приведёт к 409 на generate-final?
- [ ] **Длинные шаблоны**: удобно ли работать с деревом из ~12+ разделов (скролл, плотность,
      нет ли «простыни»)? Нужны ли свернуть/развернуть, выбрать всё/снять всё?
- [ ] **Режим default vs reference**: понятно ли, чем «Стандартная структура» отличается от
      «Из референса»? Очевиден ли шаг «Декомпозировать» и что делать с результатом?
- [ ] **Stepper — понятность статусов**: читаются ли статусы шагов (`done/ready/needs_action/`
      `blocked/processing`) без легенды? Понятно ли, какой шаг следующий?
- [ ] **Stepper на мобильном**: на узком экране шаги переносятся/не ломают layout; не нужен ли
      горизонтальный скролл вместо переноса при большем числе шагов?
- [ ] **Auto-transition между шагами**: сейчас переход ручной (Назад/Далее + клик). Нужен ли
      авто-переход к следующему действенному шагу после успешного действия?
- [ ] **Collapse/summary для done-шагов**: достаточно ли степпера, или нужны свёрнутые
      сводки завершённых шагов (что именно сделано), чтобы не открывать каждый шаг заново?
- [ ] **Редирект `/brief/:id` → `/review`**: не сбивает ли с толку, что freeform-бриф
      перекидывает с редактора на review-экран?
- [ ] **Устаревание**: заметно ли в UI, что документ устарел после изменения структуры?
- [ ] **Copy / Download**: удобны ли действия copy / download markdown / json / docx,
      понятны ли их подписи?
